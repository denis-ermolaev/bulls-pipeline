```plaintext
тип[необязательная область]: краткое описание

[необязательное тело]

[необязательная сноска(и)]
```

*   **Тип** – обязательный. Показывает, что именно сделано.
*   **Область** – необязательно. Указывает на модуль, компонент или файл (например, `auth`, `ui`, `api`).
*   **Краткое описание** – начинается с маленькой буквы, без точки в конце. Длина — до 72 символов.
*   **Тело** – подробное объяснение, если необходимо.
*   **Сноски** – например, `Closes #42`, или `BREAKING CHANGE: описание ломающего изменения`.

### Основные типы коммитов

| Тип        | Назначение                                                       |
| ------------| ------------------------------------------------------------------|
| `feat`     | Новая функциональность (фича)                                    |
| `fix`      | Исправление ошибки                                               |
| `docs`     | Только изменения в документации                                  |
| `style`    | Форматирование кода, отступы, точки с запятой (не меняет логику) |
| `refactor` | Рефакторинг кода – ни исправление, ни фича                       |
| `perf`     | Улучшение производительности                                     |
| `test`     | Добавление или исправление тестов                                |
| `chore`    | Обслуживание: обновление зависимостей, скрипты сборки            |
| `build`    | Изменения в системе сборки или внешних зависимостях              |
| `ci`       | Настройка CI/CD                                                  |
| `revert`   | Отмена предыдущего коммита                                       |

### Примеры хороших сообщений

- feat(auth): добавить вход через Яндекс ID
- fix(api): исправить падение при пустом поле email
- docs: обновить раздел по локальной разработке
- style: привести отступы к единому стилю
- refactor(cart): вынести логику расчёта скидки
- perf(db): оптимизировать запрос списка товаров
- test(cart): покрыть расчёт скидки юнит-тестами
- chore: обновить зависимости до последних стабильных версий
- ci: добавить шаг проверки форматирования

```bash
git commit -m "feat(vcf): добавить парсинг мультиаллельных вариантов"

# После второй m идёт тело коммита
git commit -m "fix(gwas): исправить расчёт lambda GC" -m "Раньше коэффициент инфляции считался по медиане хи-квадрат, что занижало значение. Перешёл на регрессионный метод."

# Открывает редактор
git commit


# feat(imputation): **реализовать** конвейер импутации через Beagle 5.4

# Подготовка включает:
# - рекодирование в формат bref3
# - разбиение хромосом на чанки по 5 Мб
# - запуск Beagle с референсной панелью ARS-UCD1.2

# BREAKING CHANGE: изменён формат выходных данных – теперь VCF вместо PLINK .raw
# Closes #12

```


### GitHub Flow

```bash
# 1. Старт
git checkout main
git pull origin main
git checkout -b feat/new-plot

# 2. Работа
# ...редактируем файлы...
git add .
git commit -m "feat(plots): добавить манхэттенский график"

# 3. Пуш и PR
git push -u origin feat/new-plot
# Создаём Pull Request на GitHub (или через gh pr create)

# 4. После одобрения (или самопроверки) нажимаем "Squash and merge" на сайте

# 5. Очистка
git checkout main
git pull origin main
git branch -d feat/new-plot
git push origin --delete feat/new-plot
# git remote prune origin — удалит локальные ссылки на ветки, которые уже удалены на GitHub
```

### Отмена изменений
Если вы уже запушили плохие коммиты – не используйте reset, лучше revert:
```bash
# Создать новый коммит, отменяющий изменения последнего коммита (безопасно для истории)
git revert HEAD

# Отменить несколько коммитов
git revert HEAD~3..HEAD
```

Для локального сброса коммитов:
```bash
# Отменить последние 2 коммита, оставив все изменения в индексе (staging)
git reset --soft HEAD~2

# Снять всё с индекса
git reset HEAD
```

```bash
# Добавить забытые изменения в последний коммит
git add забытый-файл
git commit --amend --no-edit   # не меняя сообщение

# Изменить сообщение последнего коммита
git commit --amend -m "Новое сообщение"

# Изменить и сообщение, и содержимое (добавив правки)
git add .
git commit --amend
```

Удаление файлов из репозитория, после ,например, попадания в .gitignore НО если они уже отслеживаются
```bash
git rm --cached <файл>
git rm -r --cached <папка>
# Или удалить все, добавить все
git rm -r --cached .
git add .
```
